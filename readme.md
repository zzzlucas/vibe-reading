# 技术架构与资源限制说明 (Architecture & Resource Limits)

本项目旨在提供高装饰性、纯本地流转的阅读与 AI 伪装交互体验。考虑到项目运行成本与云端依赖，以下整理了项目的核心技术栈栈以及依托于 Vercel 免费版 (Hobby Plan) 部署时的各项红线指标。

## 1. 核心技术架构
- **核心框架**：Vue 3 组合式 API + Vite 工具链 + Less 预处理器。
- **状态管理与存储层**：业务数据（小说文本、聊天缓存、深色配置等）高度依赖浏览器的 **LocalStorage** 和 **IndexedDB**，实现“数据不离本地”、 0 延迟、完全免费的客户端重型计算体验。
- **大模型通讯侧**：前端通过直连 `API Key` 的方式，通过 `fetch` (SSE流) 对接各大 LLM 提供商，确保原生的思维链 `<details>` 折叠及极速流式返回。
- **托管与云服务**：部署在 **Vercel (Hobby 免费版)**。对于极少数需要中心化验权（例如全局验证码、核心鉴权）的环节，适度剥离给 Vercel Serverless Functions 及外部轻量级 KV 存储。

---

## 2. Vercel 免费版 (Hobby) 部署限制一览
> 以下限制数据是极其准确的，这决定了服务端绝对不能做重型操作，必须时刻坚持**“能放前端算的绝不上云”**的原则。

### 核心计算与流量限制 (Vercel Functions / Edge)
*   **云函数调用次数**：100,000 次 / 月（*超出会产生警告甚至熔断停机*）。
*   **云函数执行超时**：**10 秒 / 次**（*极其严苛，绝不能跑耗时任务，只能做毫秒级的状态校验和短流控制*）。
*   **公网出站流量**：100 GB / 月（*对于纯跑文本和 JSON 返回来说，可以说是极其充足，不用担心*）。

### Vercel KV (Upstash Redis) 限制
*这通常用于保存邀请码、卡密验证状态等轻量级全局状态。*
*   **实例数量**：仅限 **1** 个免费数据库（*强依赖于 `dev:` / `prod:` 前缀来在代码里手动隔离环境*）。
*   **请求次数**：**30,000 次 / 月**（*平均每天约 1,000 次，这是最容易被打爆的红线！必须通过本地严格的防抖与 IndexedDB 缓存进行前置拦截*）。
*   **数据传输量**：256 MB / 月。
*   **存储容量**：30 MB（*存卡密和 Hash 非常充足，大概能存十几万条，但**绝不能存日志 / 用户侧的大文本***）。

### 其他辅助组件限制 (应对未来扩展使用)
*   **Vercel Postgres (关系型数据库)**：1 个数据库实例，256 MB 存储，每月 60 小时的 CPU 计算时间。
*   **Vercel Blob (简单的对象/文件存储)**：250 MB 存储容量上限，每月允许 100 万次读取，50,000 次写入操作。
